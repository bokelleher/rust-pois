# SESAME Per-Request Overhead — Measured Benchmarks

Criterion benchmarks for the `rust-pois` SESAME (SCTE 130-9) reference
implementation. Reproduce with:

```bash
cargo bench --bench sesame_overhead && python3 scripts/bench_to_md.py
```

The tables are generated by `scripts/bench_to_md.py` from Criterion's
`estimates.json` (median → **p50**) and the per-sample distribution in
`sample.json` (99th percentile → **p99**). p99 is the 99th percentile of
Criterion's per-sample iteration means — a methodology-consistent stand-in for
the per-request tail, not a production SLA.

---

## Primary result — AWS c6i.xlarge (the hardware the paper cites)

Measured on the exact instance class referenced by §9.2.

| Property | Value |
|---|---|
| Instance | AWS **c6i.xlarge** (4 vCPU) |
| CPU | Intel Xeon Platinum 8375C @ 2.90 GHz (Ice Lake) |
| Crypto acceleration | **AES-NI: yes, SHA extensions: yes** |
| OS | Ubuntu 24.04 LTS |
| Toolchain | rustc 1.96.0, `--release` |
| Criterion settings | `--warm-up-time 1 --measurement-time 3` (p50≈p99 confirms negligible variance) |
| Crypto crates | RustCrypto `hmac` 0.12, `sha2` 0.10, `aes-gcm` 0.10 (pure-Rust) |

### Per-operation overhead

| Operation | 1 KB p50 | 1 KB p99 | 4 KB p50 | 4 KB p99 | 16 KB p50 | 16 KB p99 |
|---|---|---|---|---|---|---|
| Tier 1 — HMAC sign | 0.39 µs | 0.40 µs | 0.39 µs | 0.39 µs | 0.39 µs | 0.40 µs |
| Tier 1 — verify (body hash + HMAC) | 1.37 µs | 1.39 µs | 3.68 µs | 3.75 µs | 0.013 ms | 0.013 ms |
| Tier 2 — scope authorization | 0.06 µs | 0.07 µs | 0.06 µs | 0.07 µs | 0.06 µs | 0.07 µs |
| Tier 3 — AES-256-GCM seal | 1.10 µs | 1.13 µs | 3.54 µs | 3.56 µs | 0.013 ms | 0.013 ms |
| Tier 3 — AES-256-GCM open | 1.11 µs | 1.17 µs | 3.55 µs | 3.57 µs | 0.013 ms | 0.013 ms |

### Combined per-request paths (the figures the paper should cite)

| Path | 1 KB p50 | 1 KB p99 | 4 KB p50 | 4 KB p99 | 16 KB p50 | 16 KB p99 |
|---|---|---|---|---|---|---|
| **Inbound** — verify + authz + decrypt (Tier 1+2+3) | 3.31 µs | 3.43 µs | 8.02 µs | 8.17 µs | 0.027 ms | 0.027 ms |
| **Outbound** — encrypt + sign (Tier 1+3) | 4.23 µs | 4.25 µs | 8.94 µs | 9.57 µs | 0.028 ms | 0.028 ms |

**Result.** The worst-case combined p99 across all payload sizes is **0.028 ms**
(28 µs, outbound at 16 KB) — roughly **35× inside** the §7 budget of ≤ 1 ms. The
sub-millisecond claim is validated with very large margin on the cited hardware.

> **Note for §9.2 ([BO-9]).** The draft's current §9.2 table (Tier 1 verify
> 0.04/0.08 ms, Tier 2 0.05/0.09 ms, Tier 3 +decrypt 0.32/0.51 ms) predates any
> implementation and does not match measurement — the real Tier-3 decrypt at
> 16 KB is **0.013 ms**, ~25–40× faster than the figure printed in the draft.
> Replace §9.2 with the two tables above (and this hardware row). The numbers
> are *better* than the paper claims, so this strengthens the result.

## Portability floor — QEMU VM without crypto acceleration

The same benchmark on a QEMU virtual CPU **without** AES-NI/SHA extensions
(software AES & SHA) — a conservative lower bound showing SESAME stays
sub-millisecond even with no hardware crypto, supporting the deploy-anywhere
thesis.

| Property | Value |
|---|---|
| CPU | QEMU Virtual CPU v2.5+ (16 vCPU), **no AES-NI / SHA extensions** |
| OS | Ubuntu 24.04 LTS, kernel 6.8.0-117 |
| Toolchain | rustc 1.96.0, `--release` |

| Path | 1 KB p50 | 1 KB p99 | 4 KB p50 | 4 KB p99 | 16 KB p50 | 16 KB p99 |
|---|---|---|---|---|---|---|
| Inbound — verify + authz + decrypt | 0.015 ms | 0.020 ms | 0.042 ms | 0.045 ms | 0.157 ms | 0.180 ms |
| Outbound — encrypt + sign | 0.017 ms | 0.019 ms | 0.046 ms | 0.055 ms | 0.162 ms | 0.219 ms |

Even with no crypto hardware, worst-case combined p99 is **0.219 ms** — still
comfortably under 1 ms. Hardware AES-NI/SHA (the c6i numbers above) cuts this by
roughly 8×.

## Reproduce

Locally:

```bash
cargo bench --bench sesame_overhead && python3 scripts/bench_to_md.py
```

On a fresh c6i.xlarge (launches, benchmarks, prints the table, terminates):

```bash
scripts/run_c6i_bench.sh --instance-profile <ssm-instance-profile> --git-ref <branch>
```

See the header of `scripts/run_c6i_bench.sh` for the IAM policy and instance
profile setup. (The c6i numbers above were produced this way — Ubuntu 24.04,
SSM-driven, instance auto-terminated.)

Notes:
* Tier 1 sign is payload-size-independent (it hashes only the canonical string);
  Tier 1 verify and Tier 3 seal/open scale with payload size as body hashing /
  GCM dominate.
* On c6i, p50 ≈ p99 throughout — variance is negligible at these magnitudes.
