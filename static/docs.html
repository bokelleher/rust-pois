For users & integrators

Overview

POIS is an ESAM (Event Signaling and Management, SCTE 130-3) server that processes SCTE-35 signals and applies rule-based conditioning. It provides:

  • ESAM Signal Processing - Handle SignalProcessingEvent XML, return a SignalProcessingNotification
  • Rule-Based Conditioning - Match by command, segmentation type/UPID, signal ID glob, or UTC window; then pass-through, replace, delete, blackout, regionalize, shorten/extend/fill, slate, or tracking-only
  • Faithful SCTE-35 editing - In-place rewrite of UPID, break duration, and delivery-restriction flags (lengths + CRC recomputed), plus a built/literal payload builder
  • Decision metadata - A standard ESAM verb on the wire (replace/delete/noop) plus an optional <pois:Decision> element carrying the friendly action + params for a downstream packager
  • SESAME security (SCTE 130-9) - Optional HMAC auth, channel-scoped authorization, and AES-256-GCM payload encryption over the ESAM exchange (see below)
  • SCTE-35 tools - Build / decode / validate (base64, hex, or binary input) — see the Tools page
  • Multi-Channel + Event Logging + RESTful API - Independent rule sets per channel, full event history, and management endpoints
  • Projects & Templates - Capture a channel (or a single rule) as a reusable template, group templates into a project, and apply them to spin up new channels

Multi-tenancy. Channels, rules, events, projects, and templates are scoped by group membership. Every /api/* response is filtered to what you are allowed to see, so two stations using the same server never see each other's resources. You can manage what you own and any resource shared to a group you administer. The group and sharing administration endpoints live in the Admin Guide (visible to admins); this guide covers the day-to-day user and integrator surface.

Authentication

Every management endpoint (everything under /api/*, except /api/auth/login) requires a Bearer token:

Authorization: Bearer <token>

Logging in (session token)

The primary way to get a token is to log in with your username and password. The response carries a JSON Web Token (your session token) plus your profile, including the groups you belong to and your per-group role.

POST /api/auth/login

Authenticate and receive a session token. This endpoint is public (no Bearer token needed).

Request Body

{ "username": "wanf.op", "password": "••••••••" }

Response

{ "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "user": { "id": 7, "username": "wanf.op", "role": "user", "enabled": true, "email": null, "created_at": "2026-05-01T12:00:00Z", "last_login": "2026-06-05T09:30:00Z", "must_change_password": false, "groups": [ { "id": 3, "name": "WANF-Atlanta", "role": "member" } ] } }

Send Authorization: Bearer <token> on every subsequent request. To check who you are (and refresh your group list), call GET /api/auth/me, which returns the same user object.

First-login note. If your account was just provisioned (or an admin reset your password), must_change_password is true and the server returns 403 on every endpoint except POST /api/auth/change-password until you set your own password. See Change password.

API tokens (long-lived)

For unattended integrations you can mint a named API token instead of logging in interactively. Manage these on the Tokens page or via the API:

GET POST /api/tokens

List your tokens, or create one: { "name": "my-integration", "expires_in_days": 90 } (omit expires_in_days for no expiry). The plaintext token is returned once, at creation. Revoke with DELETE /api/tokens/{id}.

Both token types are sent the same way (Authorization: Bearer …) and resolve to your identity and groups live on each request. If you only need short-lived access, the login session token is the simplest path.

ESAM is separate

The ESAM endpoint (/esam) does not use Bearer tokens. By default it is unauthenticated (regular SCTE 130-3 ESAM); per-channel SESAME (SCTE 130-9) can instead require a minimum security tier — HMAC-SHA256 auth (Tier 1), channel-scoped authorization (Tier 2), and AES-256-GCM payload encryption (Tier 3) — carried in HTTP headers with no ESAM XML schema change. Responses are signed (and, for Tier 3, encrypted) regardless, so a consumer can verify the POIS decision wasn't forged. SESAME configuration is covered in the Admin Guide.

For operators

Using the Web UI

Most day-to-day tasks need no API calls. After logging in you can do everything below from the browser; the pages call the same endpoints documented here and honor the same group scoping.

  • Channels & Rules - create channels, add and reorder rules with the rule editor (drag to reorder), and toggle a channel on or off. When you create a channel you pick which of your groups it belongs to (if you are in more than one).
  • Event Monitor - browse processed ESAM events, filter by channel / action / time, and free-text search by acquisition signal ID, source IP, SCTE-35 command, or UPID. The SESAME column shows the achieved security tier per event.
  • Projects & Templates - save a channel or a single rule as a reusable template, organize templates into projects, and apply a template or a whole project to spin up new channels.
  • SCTE-35 Tools - build, decode, and validate SCTE-35 payloads (base64, hex, or binary) without sending a live signal.
  • Your password - change your own password at any time from the account menu.

Common Use Cases

Here are practical examples of how to use POIS for real-world scenarios:

1
Convert Program End to Ad Break
Scenario: Your playout system marks the end of a program with a Program End segmentation descriptor (type 0x11), but your ad insertion system expects a splice_insert ad-out for breaks.

Solution: Match the Program End descriptor and replace it with a freshly built 60-second splice_insert ad-out:

Name:
Program End → 60s Ad Break
Action:
replace
Match:
{"anyOf":[{"scte35.segmentation_type_id":"0x11"}]}
Parameters:
{"build":{"command":"splice_insert_out","duration_s":60}}

Result: Every Program End signal is converted to a 60-second ad opportunity.

2
Filter Out Internal Test Signals
Scenario: Your testing team sends SCTE-35 signals with "test-" prefixed IDs that should not reach production ad servers.

Solution: Create a high-priority rule to delete any signals with test IDs:

Name:
Delete Test Signals
Priority:
1 (highest)
Action:
delete
Match:
{"anyOf":[{"acquisitionSignalID":"test-*"}]}

Result: All signals with IDs starting with "test-" are dropped (no BinaryData emitted) before reaching downstream systems.

3
Adjust Ad Break Duration
Scenario: Your network requires all ad breaks to be exactly 90 seconds, but upstream signals vary between 60-120 seconds.

Solution: Condition the incoming cue in place with shorten (or fill), setting the break duration to 90 seconds. The break duration is rewritten and the SCTE-35 length and CRC are recomputed:

Name:
Normalize to 90s Breaks
Action:
shorten
Match:
{"anyOf":[{"scte35.command":"splice_insert"}]}
Parameters:
{"to_duration_s":90}

Result: Every splice_insert break is set to exactly 90 seconds.

4
Black Out a Regional Avail
Scenario: A national feed carries an ad avail that must be regionally restricted (no web delivery, regional blackout enforced) on your affiliate.

Solution: Use blackout to set the delivery-restriction flags on the incoming cue in place (lengths and CRC are recomputed):

Name:
Regional Blackout
Action:
blackout
Match:
{"anyOf":[{"scte35.segmentation_type_id":"0x34"}]}
Parameters:
{"restrictions":{"web_delivery_allowed":false,"no_regional_blackout":false,"archive_allowed":true,"device_restrictions":0}}

Result: Each Provider Placement Opportunity (0x34) carries restriction flags so downstream packagers enforce the blackout. The friendly action is also echoed in <pois:Decision>.

5
Route Different Channels to Different Ad Servers
Scenario: You have sports content on Channel A (using AdServer1) and news on Channel B (using AdServer2), each requiring different signal formats.

Solution: Create separate channels with custom rules:

Channel A:
sports-live
Rule:
replace — build a 30s splice_insert ad-out (AdServer1 format)
Parameters:
{"build":{"command":"splice_insert_out","duration_s":30}}
Channel B:
news-24x7
Rule:
replace — build a time_signal with a Program Start descriptor (AdServer2 format)
Parameters:
{"build":{"command":"time_signal","segmentation_type_id":"0x10"}}

Result: Each channel independently transforms signals according to its downstream ad server requirements.

# Sports channel uses sports-live rules POST /esam?channel=sports-live # News channel uses news-24x7 rules POST /esam?channel=news-24x7

ESAM Endpoint

POST /esam?channel={channel_name}

Process SCTE-35 signal through channel rules

Query Parameters

Parameter Type Required Description
channel string Yes Channel name to use for processing

Request Body

XML SignalProcessingEvent. The SCTE-35 payload is carried as base64 in <BinaryData signalType="SCTE35">; acquisitionPointIdentity typically names the channel.

<?xml version="1.0"?> <SignalProcessingEvent xmlns="urn:cablelabs:iptvservices:esam:xsd:signal:1"> <AcquiredSignal acquisitionSignalID="test-001" acquisitionPointIdentity="sportsfeed-east"> <UTCPoint utcPoint="2026-06-02T20:00:00Z"/> <BinaryData signalType="SCTE35">/DAhAAAAAAAAAP/wEAUAAAAAf+//AAAAAH4AAAAAAAA1FwHt</BinaryData> </AcquiredSignal> </SignalProcessingEvent>

Response

XML SignalProcessingNotification. The ResponseSignal action attribute is always a standard ESAM verb (replace, delete, or noop); the conditioned SCTE-35 is returned in <sig:BinaryData> (omitted for delete). When a rule matches, an optional <pois:Decision> element echoes the friendly action and its params for a downstream packager (ignored by stock encoders).

<?xml version="1.0"?> <SignalProcessingNotification xmlns="urn:cablelabs:iptvservices:esam:xsd:signal:1" xmlns:sig="urn:cablelabs:md:xsd:signaling:3.0" xmlns:core="urn:cablelabs:md:xsd:core:3.0" xmlns:common="urn:cablelabs:iptvservices:esam:xsd:common:1" xmlns:pois="urn:techex:pois:decision:1"> <common:StatusCode classCode="0"><core:Note>replaced signal</core:Note></common:StatusCode> <ResponseSignal action="replace" acquisitionSignalID="test-001" acquisitionPointIdentity="sportsfeed-east"> <sig:UTCPoint utcPoint="2026-06-02T20:00:00Z"/> <sig:BinaryData signalType="SCTE35">/DBLAAAAAAAAAP/wEAUAAAAB...</sig:BinaryData> <pois:Decision action="blackout">{"restrictions":{"no_regional_blackout":false}}</pois:Decision> </ResponseSignal> </SignalProcessingNotification>

Alternative URL Format

Channel can also be specified as a path parameter:

POST /esam/channel/{channel_name}

Channels API

A channel owns an independent rule set. The list is scoped to the channels you may see (owned, global, or shared to one of your groups). Note on identifiers: the ESAM endpoint addresses channels by their unique name, but the management API addresses a single channel by its numeric id. Channel names are globally unique.

enabled and is_global are returned as integers (0/1). owner_user_id is the id of the creating user (may be null for seeded channels). There is no single-channel GET; read a channel from the list response.

Rules API

GET /api/channels/{channel_id}/rules

List all rules for a channel, by numeric channel id (ordered by priority). Requires read access to the channel.

Response

[ { "id": 1, "channel_id": 15, "name": "Delete splice_insert", "priority": 10, "enabled": 1, "match_json": "{\"anyOf\":[{\"scte35.command\":\"splice_insert\"}]}", "action": "delete", "params_json": "{}", "owner_user_id": 3, "deleted_at": null, "created_at": "2026-05-01T10:00:00Z", "updated_at": "2026-05-01T10:00:00Z" } ]

enabled is an integer (0/1). match_json and params_json are returned as JSON strings (see the create-rule note below).

POST /api/channels/{channel_id}/rules

Create a new rule on a channel, by numeric channel id (priority -1 appends at end, spaced by 10). Requires write access to the channel.

Request Body

{ "name": "Delete splice_insert", "priority": -1, "enabled": true, "action": "delete", "match_json": { "anyOf": [ { "scte35.command": "splice_insert" } ] }, "params_json": {} }

Important: on the way in, match_json and params_json are JSON objects (not stringified JSON). The server stores them as strings, so list/get responses return them as strings. Pass priority: -1 to append the rule at the end.

Rule Actions

The action is a friendly label; on the ESAM wire it maps to a standard verb and the effect is carried in the conditioned SCTE-35 (plus <pois:Decision> metadata).

Action ESAM verb Effect
noopnoopPass the original signal through unchanged
replacereplaceReplace with a built / literal payload, or rewrite the incoming UPID
deletedeleteDrop the signal (no BinaryData emitted)
blackoutreplaceSet delivery-restriction flags on the incoming cue
regionalizereplaceRewrite the incoming UPID (regional/affiliate targeting)
shortenreplaceSet the incoming break to a target duration (to_duration_s)
extendreplaceAdd to the incoming break duration (duration_s, additive)
fillreplaceSet the break to a target duration (to_duration_s; filler is a packager hint)
slatenoopPass through + signal the MCR/packager to insert a slate
tracking-onlynoopPass through + report only (no ad decisioning)

Unknown actions fall back safely to noop. blackout/regionalize/shorten/extend/fill edit the incoming cue in place (lengths + CRC are recomputed); if there is no cue to condition the original is passed through.

Match Criteria (match_json)

Combine leaf conditions with anyOf (OR) and/or allOf (AND). An empty object {} is a deliberate catch-all. Supported leaf keys:

KeyMatch
acquisitionSignalIDGlob (* wildcards, e.g. blk-*, *AFE1*)
scte35.commandCommand name, case-insensitive (splice_insert, time_signal, …)
scte35.segmentation_type_idHex id, e.g. 0x34
scte35.segmentation_upidGlob against the decoded UPID
utcBetween{"start":"…Z","end":"…Z"} ISO-8601 UTC window
// ANY splice_insert OR a Program Start/End segmentation descriptor {"anyOf":[{"scte35.command":"splice_insert"},{"scte35.segmentation_type_id":"0x10"}]} // ALL of: signal id glob AND a prime-time window {"allOf":[{"acquisitionSignalID":"prod-*"},{"utcBetween":{"start":"2026-06-02T18:00:00Z","end":"2026-06-02T23:00:00Z"}}]} // Catch-all (matches every signal) {}

Action Parameters (params_json)

// replace — build a fresh avail (splice_insert ad-out), optional segmentation {"build":{"command":"splice_insert_out","duration_s":30, "segmentation_type_id":"0x34","upid_type":"0x0C","upid":"ABCD1234"}} // replace — build an immediate time_signal {"build":{"command":"time_signal_immediate"}} // replace — literal SCTE-35 payload (base64) {"scte35_b64":"/DAh..."} // replace / regionalize — rewrite the incoming UPID in place (type optional = keep) {"rewrite_upid":{"upid":"region-east-001","upid_type":"0x0C"}} // (replace) {"upid":"region-east-001","upid_type":"0x0C","region":"us-east"} // (regionalize) // blackout — set delivery-restriction flags {"restrictions":{"web_delivery_allowed":false,"no_regional_blackout":false, "archive_allowed":true,"device_restrictions":0}} // shorten / fill — SET the break to a target duration (seconds) {"to_duration_s":30,"filler":"house"} // extend — ADD seconds to the incoming break duration (additive) {"duration_s":15} // slate — packager hint; noop / delete / tracking-only — no params {"asset":"slate-1","duration_s":10} {}
PUT /api/rules/{rule_id}

Update an existing rule (full body, same fields as create). Requires write access to the rule's channel.

DELETE /api/rules/{rule_id}

Soft-delete a rule. Requires write access to the rule's channel.

POST /api/rules/reorder

Reorder rules by priority — body: {"ordered_ids":[3,1,2]} (reassigns priorities 0,10,20,… in that order). All ids must belong to one channel you can write; returns 204 No Content.

Projects & Templates

A template captures a single rule (kind: "rule") or a whole channel and its rules (kind: "channel") as a reusable snapshot. A project groups templates together. Applying a channel template (or a project) spins up new channels; applying a rule template appends the rule to an existing channel. Like everything else, projects and templates are group-scoped, and their list/detail responses carry a group_ids array indicating which groups they are shared to.

Projects

Events API

Every ESAM exchange is logged. Event queries are scoped to channels you may read.

GET /api/events

Query processed events (newest first)

Query Parameters

Parameter Type Default Description
limit integer 100 Number of events to return
offset integer 0 Skip this many events (pagination)
channel string - Filter by channel name
action string - Filter by the rule action recorded on the event (e.g. noop, replace, delete, blackout, shorten, tracking-only)
since string - Lower bound on the event timestamp (ISO-8601 / SQL datetime, e.g. 2026-06-05T00:00:00Z). Events with timestamp >= since are returned.
search string - Free-text match across acquisition signal ID, source IP, SCTE-35 command, and UPID (an ASCII term is also matched against the hex-encoded UPID)

Response

[ { "id": 123, "timestamp": "2026-06-05T20:00:00Z", "channel_name": "WANF-Primary", "acquisition_signal_id": "test-001", "utc_point": "2026-06-05T20:00:00Z", "source_ip": "203.0.113.7", "scte35_command": "splice_insert", "scte35_type_id": "52", "scte35_upid": "0x0C:41424344", "scte35_b64": "/DBLAAAAAAAAAP/wEAUAAAAB...", "matched_rule_id": 7, "matched_rule_name": "Regional Blackout", "action": "blackout", "processing_time_ms": 2, "response_status": 200, "error_message": null, "sesame_tier": 3, "channel_timezone": "America/New_York", "rule_priority": 10 } ]

For administrators

Admin Overview

POIS is multi-tenant. Resources (channels, projects, templates) are owned by a user and shared to groups; rules and ESAM events inherit their channel's groups. Two kinds of elevated access exist:

  • Super-admin (users.role = "admin") — bypasses all group checks; manages groups, users, and org-wide (global) visibility.
  • Group-admin (group_members.role = "admin" for a group) — manages membership and content within that group only. This is a per-membership role, not a user role.

The access model (super-admin bypasses everything):

  • Read a resource ⇔ you own it, OR it is global, OR it is shared to a group you are a member of.
  • Write (edit / delete / share) ⇔ you own it, OR you are a group-admin of a group it is shared to.
For administrators

Groups & RBAC

A group is a tenant boundary (e.g. a station or a national channel). Membership carries a per-group role of admin or member. Identity is resolved live from the database on every request, so membership changes take effect immediately (no need to re-issue tokens).

GET /api/me/groups

The caller's own groups + per-group role, and whether they are a super-admin. Used by the UI to gate admin features.

Response

{ "super_admin": false, "groups": [ { "id": 3, "name": "WANF-Atlanta", "role": "admin" } ] }
GET /api/groups POST

List groups (super-admin: all; otherwise your groups). POST creates a group (super-admin only). { name, description? }

GET PUT DELETE /api/groups/{id}

Get a group + members (member), rename / edit (group-admin), or soft-delete (super-admin).

GET POST /api/groups/{id}/members

List members, or add / change a member's role (group-admin). { user_id, role: "admin" | "member" }. Remove with DELETE /api/groups/{id}/members/{user_id}.

For administrators

Users & Passwords

Super-admins manage all users; group-admins manage users within the groups they administer (created users are auto-added to that group with role user — a group-admin can never mint a super-admin).

GET POST /api/users

List users in scope, or provision a new one. { username, password, role, email?, group_id? }. A provisioned account is created with a temporary password and must_change_password set.

PUT DELETE /api/users/{id}

Update role / enabled / email / password (admin reset re-arms must_change_password), or delete (super-admin; the bootstrap admin id 1 is protected).

Forced password change on first login

When an account is flagged must_change_password (every API-provisioned account, and any account whose password an admin resets), the server blocks every endpoint with 403 except the change-password one until the user sets their own password. API tokens are exempt.

POST /api/auth/change-password

Set a new password. { current_password, new_password }. The new password must differ from the current one and be at least 8 characters; success clears the forced-change flag.

For administrators

Resource Sharing

Share an existing channel, project, or template to specific groups. A non-super caller may only target groups they belong to; shares to groups outside their reach are preserved (a scoped merge), so a group-admin can never clobber another tenant's access. Only super-admins may toggle org-wide (global) visibility.

GET PUT /api/share/{kind}/{id}

kind is channel, project, or template. GET returns current sharing; PUT applies it.

GET Response

{ "kind": "channel", "id": 15, "group_ids": [3], "is_global": false, "can_write": true }

PUT Request Body

{ "group_ids": [3, 5], "is_global": false }
For administrators

SESAME Security (SCTE 130-9)

SESAME secures the ESAM exchange. It is configured per deployment via POIS_SESAME_* environment variables (key material in POIS_SESAME_KEYS) and enforced per channel via channels.sesame_min_tier:

  • Tier 1 — HMAC-SHA256 request authentication.
  • Tier 2 — channel-scoped authorization (a key may only act on its channels).
  • Tier 3 — AES-256-GCM payload encryption (encrypt-then-MAC).

Responses are always signed (and, at Tier 3, encrypted) so a consumer can verify the POIS decision was not forged. A channel with a minimum tier rejects requests below it with 401. The achieved tier is recorded per event and shown in the Event Monitor's SESAME column. With no POIS_SESAME_* config the ESAM path is unauthenticated (legacy SCTE 130-3 behavior).
