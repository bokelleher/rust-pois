Overview

POIS is an ESAM (Event Signaling and Management) server that processes SCTE-35 signals and applies rule-based transformations. It provides:

  • ESAM Signal Processing - Handle SignalProcessingEvent XML requests
  • Rule-Based Transformations - Match, modify, replace, or delete signals
  • Multi-Channel Support - Independent rule sets per channel
  • Event Logging - Track all processed signals
  • RESTful API - Manage channels, rules, and view events

Authentication

All API endpoints (except ESAM) require Bearer token authentication:

Authorization: Bearer your-token-here

The ESAM endpoint (/esam) does not require authentication.

Getting a Token

Tokens are managed through the web interface or API:

  • Web UI: Tokens Page
  • API: POST /api/tokens

Common Use Cases

Here are practical examples of how to use POIS for real-world scenarios:

1
Convert Program End to Ad Break
Scenario: Your playout system sends program_end signals, but your ad insertion system expects splice_insert for ad breaks.

Solution: Create a rule that transforms program_end to splice_insert with a 60-second break duration:

Name:
Program End → 60s Ad Break
Action:
replace
Match:
{"anyOf":[{"scte35.command":"program_end"}]}
Parameters:
{"command":"splice_insert","duration_ticks":5400000,"pts_adjustment":0}

Result: Every program_end signal is automatically converted to a 60-second ad opportunity.

2
Filter Out Internal Test Signals
Scenario: Your testing team sends SCTE-35 signals with "test-" prefixed IDs that should not reach production ad servers.

Solution: Create a high-priority rule to delete any signals with test IDs:

Name:
Delete Test Signals
Priority:
1 (highest)
Action:
delete
Match:
{"anyOf":[{"acquisition_signal_id":"test-*"}]}

Result: All signals with IDs starting with "test-" are silently dropped before reaching downstream systems.

3
Adjust Ad Break Duration
Scenario: Your network requires all ad breaks to be exactly 90 seconds, but upstream signals vary between 60-120 seconds.

Solution: Create a rule that normalizes all splice_insert durations to 90 seconds:

Name:
Normalize to 90s Breaks
Action:
replace
Match:
{"anyOf":[{"scte35.command":"splice_insert"}]}
Parameters:
{"command":"splice_insert","duration_ticks":8100000}

Result: All splice_insert signals are rewritten to have exactly 90 seconds duration (8.1M ticks at 90kHz).

4
Add PTS Adjustment for Encoding Delay
Scenario: Your encoding pipeline introduces a 2-second delay, and SCTE-35 timing needs to be adjusted to compensate.

Solution: Create a rule that adds 180,000 ticks (2 seconds at 90kHz) to all signals:

Name:
Add 2s Encoding Delay
Action:
replace
Match:
{"anyOf":[{}]}
Parameters:
{"pts_adjustment":180000}

Result: All SCTE-35 signals have their timing adjusted by 2 seconds to synchronize with the encoding delay.

5
Route Different Channels to Different Ad Servers
Scenario: You have sports content on Channel A (using AdServer1) and news on Channel B (using AdServer2), each requiring different signal formats.

Solution: Create separate channels with custom rules:

Channel A:
sports-live
Rule:
Convert all to 30s breaks (AdServer1 format)
Parameters:
{"command":"splice_insert","duration_ticks":2700000}
Channel B:
news-24x7
Rule:
Convert all to program_start/program_end (AdServer2 format)
Parameters:
{"command":"program_start"}

Result: Each channel independently transforms signals according to its downstream ad server requirements.

# Sports channel uses sports-live rules POST /esam?channel=sports-live # News channel uses news-24x7 rules POST /esam?channel=news-24x7

ESAM Endpoint

POST /esam?channel={channel_name}

Process SCTE-35 signal through channel rules

Query Parameters

Parameter Type Required Description
channel string Yes Channel name to use for processing

Request Body

XML SignalProcessingEvent conforming to SCTE-130 Part 5

<?xml version="1.0"?> <SignalProcessingEvent xmlns="urn:cablelabs:iptvservices:esam:xsd:signal:1"> <AcquiredSignal acquisitionSignalID="test-001"> <UTCPoint utcPoint="2025-11-05T20:00:00Z"/> <SCTE35PointDescriptor> <SCTE35Command>/DA4AAAAAAAA///wBQb+AAAAAAA=</SCTE35Command> </SCTE35PointDescriptor> </AcquiredSignal> </SignalProcessingEvent>

Response

XML SignalProcessingNotification with processed signal

<?xml version="1.0"?> <SignalProcessingNotification xmlns="urn:cablelabs:iptvservices:esam:xsd:signal:1"> <ResponseSignal acquisitionSignalID="test-001"> <UTCPoint utcPoint="2025-11-05T20:00:00Z"/> <SCTE35PointDescriptor> <SCTE35Command>/DBHAAAAAAAA///wBQb+AAAAAAA=</SCTE35Command> </SCTE35PointDescriptor> </ResponseSignal> </SignalProcessingNotification>

Alternative URL Format

Channel can also be specified as a path parameter:

POST /esam/channel/{channel_name}

List all rules for a channel

Response

[ { "id": 1, "channel_id": 1, "name": "Delete splice_insert", "priority": 10, "enabled": true, "action": "delete", "match_json": "{\"anyOf\":[{\"scte35.command\":\"splice_insert\"}]}", "params_json": "{}", "created_at": "2025-11-01T10:00:00Z" } ]
POST /api/channels/{channel_name}/rules

Create a new rule

Request Body

{ "name": "Delete splice_insert", "priority": 10, "enabled": true, "action": "delete", "match_json": "{\"anyOf\":[{\"scte35.command\":\"splice_insert\"}]}", "params_json": "{}" }

Rule Actions

Action Description
noop Log match but don't modify signal
delete Remove signal from output
replace Modify signal based on params_json

Match Criteria

The match_json field supports pattern matching:

// Match any signal with splice_insert command {"anyOf":[{"scte35.command":"splice_insert"}]} // Match signals with specific acquisition ID {"anyOf":[{"acquisition_signal_id":"test-001"}]} // Match using wildcards {"anyOf":[{"acquisition_signal_id":"prod-*"}]} // Match all signals (empty condition) {"anyOf":[{}]}

Replace Parameters

The params_json field for replace action:

// Change command type {"command":"splice_insert"} // Set duration (in 90kHz ticks) {"duration_ticks":5400000} // Adjust PTS timing {"pts_adjustment":180000} // Combine multiple changes {"command":"splice_insert","duration_ticks":5400000,"pts_adjustment":0}
PUT /api/channels/{channel_name}/rules/{rule_id}

Update an existing rule

DELETE /api/channels/{channel_name}/rules/{rule_id}

Delete a rule

Events API

GET /api/events

Query processed events

Query Parameters

Parameter Type Default Description
limit integer 100 Number of events (max 1000)
offset integer 0 Skip this many events
channel string - Filter by channel name
action string - Filter by action (noop, delete, replace)
since string - Time period (1h, 24h, 7d, 30d)
GET /api/events/{event_id}

Get full event details including raw XML

Response

{ "id": 123, "timestamp": "2025-11-05T20:00:00Z", "channel_name": "live-sports", "acquisition_signal_id": "test-001", "action": "replace", "matched_rule_name": "Program End → 60s Ad", "raw_esam_request": "<SignalProcessingEvent>...</SignalProcessingEvent>", "raw_esam_response": "<SignalProcessingNotification>...</SignalProcessingNotification>" }

Code Examples

cURL Examples

Send ESAM Request

curl -X POST "http://localhost:8090/esam?channel=live-sports" \ -H "Content-Type: application/xml" \ -d '<?xml version="1.0"?> <SignalProcessingEvent xmlns="urn:cablelabs:iptvservices:esam:xsd:signal:1"> <AcquiredSignal acquisitionSignalID="test-001"> <UTCPoint utcPoint="2025-11-05T20:00:00Z"/> </AcquiredSignal> </SignalProcessingEvent>'

Create Channel

curl -X POST "http://localhost:8090/api/channels" \ -H "Authorization: Bearer dev-token" \ -H "Content-Type: application/json" \ -d '{"name":"live-sports","enabled":true}'

Create Rule

curl -X POST "http://localhost:8090/api/channels/live-sports/rules" \ -H "Authorization: Bearer dev-token" \ -H "Content-Type: application/json" \ -d '{ "name": "Delete splice_insert", "priority": 10, "action": "delete", "match_json": "{\"anyOf\":[{\"scte35.command\":\"splice_insert\"}]}", "params_json": "{}" }'

Get Events

curl -H "Authorization: Bearer dev-token" \ "http://localhost:8090/api/events?channel=live-sports&limit=10"

Rust Examples
